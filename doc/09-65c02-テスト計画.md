# 65C02 テスト計画

hoglet67/verilog-6502 の動作正確性を3段階で検証する。

## テスト戦略

### Level 1 — Klaus Dormann 機能テスト（最優先）

業界標準の 6502/65C02 テスト ROM。CPU が正しければ特定アドレスでトラップループに入る。
失敗すると別アドレスで止まり、どの命令グループで壊れたかがわかる。

#### 成功判定

| バリアント | 成功トラップ |
| --- | --- |
| 6502 版 | `$3469` で無限ループ |
| 65C02 版 | 同様のトラップアドレス（ROM 先頭コメント参照） |

#### 実施手順（WASM シミュレータ上）

```text
1. テスト ROM バイナリを $0000 から書き込む
2. 速度スライダーを最大にして Run
3. PC が成功トラップに到達したら PASS
4. 別アドレスで止まった場合は PC 値でどのテストが失敗したか特定
```

**実装コスト**: 低（バイナリを harness に組み込むか wozmon で流し込む）

**参照**: <https://github.com/Klaus2m5/6502_65C02_functional_tests>

### Level 2 — Tom Harte ProcessorTests（サイクル精度）

1 命令ごとに初期状態・期待最終状態・メモリアクセス列を JSON で定義した
10,000 ケース/opcode のテストスイート。

#### 検証内容

- 各サイクルのアドレスバス（AB）・データバス（DB）・WE の値
- 実行後のレジスタ（A, X, Y, SP, PC, P）
- 実行後のメモリ書き換え内容

#### 実施方針

```text
verif/tb_unit/ に C++ テストドライバを作成
  → JSON を読み込み、サイクルごとに AB/DB/WE を eval() 後に比較
  → 失敗時はサイクル番号・期待値・実測値を出力
```

**実装コスト**: 中（JSON パーサが必要、nlohmann/json 等を利用）

**参照**: <https://github.com/TomHarte/ProcessorTests/tree/main/nes6502>
（65C02 版も同リポジトリに存在）

---

### Level 3 — Integer BASIC スモークテスト

harness.cpp に Integer BASIC ROM（$E000–$EFFF）がすでに組み込まれている。
wozmon から `E000R` で起動でき、人間が目で結果を確認できる最もわかりやすいテスト。

#### 手順

```text
1. シミュレータ起動 → \ プロンプト確認
2. E000R → BASIC 起動メッセージ表示を確認
3. PRINT 2+2  → 4
4. PRINT 10*10 → 100
5. PRINT 32767+1 → オーバーフロー挙動の確認
```

**検証できること**: 乗算・除算・加算ルーチン、文字列出力、入力ループ

**実装コスト**: ゼロ（ROM 追加済み）

## 優先度まとめ

| テスト | 実装コスト | カバレッジ | 備考 |
| --- | --- | --- | --- |
| BASIC スモーク | ゼロ | 統合動作 | まず確認 |

| Tom Harte | 中 | サイクル精度 | hoglet の既知バグ洗い出しに有効 |
