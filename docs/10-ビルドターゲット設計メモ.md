# ビルドターゲット設計メモ

## ターゲット一覧と優先順位

| 優先 | ターゲット | 目的 |
| --- | --- | --- |
| ★ 1 | **Linux native sim** | Verilator → g++。デバッグ・反復最速 |
| ★ 2 | **cocotb (Python testbench)** | RTL 単体回帰テスト。native sim と同じコンテナで動く |
| ★ 3 | **WebAssembly (browser)** | Emscripten 経由。メインショーケース |
| △ 4 | **Wasm + Node.js / WASI** | ブラウザ無し CI で headless 実行（wasmtime / node） |
| △ 5 | **Pico 2 (RP2350)** | 実機フラッシュ。ロードマップ後半 |

### native → wasm の順が正しい理由

- native は `verilator --binary` → g++ のみ。ビルド数秒、gdb 使用可
- Wasm は Emscripten が挟まる分ビルドが重く、デバッガが使えない
- RTL の論理的正しさを native で確認してから Wasm に持ち込む

## Dockerfile 分離案 vs マルチステージ案

### 案 A: ターゲット別 Dockerfile（現採用方針）

```text
docker/
  Dockerfile          ← native sim（Verilator のみ、軽量）
  Dockerfile.wasm     ← Verilator + Emscripten
  Dockerfile.pico     ← Verilator + arm-gcc + Pico SDK + OpenOCD
```

#### メリット

- 各イメージが独立していて分かりやすい
- native sim コンテナが小さいので `docker pull` / `docker build` が速い
- Dockerfile.pico が壊れても他に影響しない

#### デメリット

- Verilator インストール手順が 3 ファイルに重複する
- `VERILATOR_TAG` などのバージョン変数を 3 箇所で同期する必要がある

---

### 案 B: マルチステージ Dockerfile（`--target` 切り替え）

```dockerfile
FROM ubuntu:24.04 AS base
# Verilator を 1 回だけインストール

FROM base AS sim
# native sim 用（追加なし、または cocotb）

FROM base AS wasm
# + Emscripten

FROM base AS pico
# + arm-gcc + Pico SDK + OpenOCD
```

```bash
docker build --target sim  -t sfa-sim  docker/
docker build --target wasm -t sfa-wasm docker/
docker build --target pico -t sfa-pico docker/
```

#### メリット

- Verilator のバージョン・インストール手順が 1 箇所で済む
- CI ではビルドキャッシュを `base` レイヤーで共有できる

#### デメリット

- マルチステージの仕組みを知らないと読みにくい
- ファイルが 1 本になるので、Pico 関連の大量の `RUN` がすべて見える

---

## 推奨: 案 A（ファイル分離）

### 理由

このプロジェクトは 3 ターゲットのビルドを**同時に走らせることがほぼない**。
Pico は実機ターゲットであり、CI のメインは native sim + cocotb + Wasm。

- マルチステージは「同一 CI ジョブで 3 ターゲットを並列ビルド」するときに真価を発揮する
- 今は各ターゲットを**必要なときだけビルド**するので、ファイルを分けた方が見通しが良い
- `VERILATOR_TAG` の重複は Compose の `build.args` 側で環境変数として一元管理すれば解消できる

```yaml
# compose.yml 側で一元化する例
x-verilator-version: &verilator-tag
  VERILATOR_TAG: "v5.042"

services:
  build-sim:
    build:
      context: .
      dockerfile: Dockerfile
      args:
        <<: *verilator-tag
  build-wasm:
    build:
      context: .
      dockerfile: Dockerfile.wasm
      args:
        <<: *verilator-tag
```

マルチステージへの移行は、CI で Wasm + Pico のビルドを並列化したくなった段階で検討する。
