### 2.2 Klaus Dormann 6502 機能テスト

業界標準の 6502/65C02 テスト ROM。CPU が正しければ特定アドレスでトラップループに入る。
失敗すると別アドレスで止まり、どの命令グループで壊れたかがわかる。

| 項目 | 内容 |
| --- | --- |
| リポジトリ | [Klaus2m5/6502_65C02_functional_tests](https://github.com/Klaus2m5/6502_65C02_functional_tests) |
| ライセンス | GPL v3 |
| 使用ファイル | `6502_functional_test.bin`（NMOS / 64KB フラット） |

```bash
# ローカルで直接実行
verif/dormann/run.sh

# Docker 経由（CI と同等環境）
docker compose -f docker/compose.yml run --rm dormann
```

成功時の出力:

```text
PASS: trapped at $3469 after 96241373 cycles
```

失敗時は `FAIL: trapped at $XXXX` と表示され、PC 値からどのテストが失敗したか特定できる。

#### 成功判定

| バリアント | 成功トラップ |
| --- | --- |
| 6502 版 | `$3469` で無限ループ |
| 65C02 版 | 同様のトラップアドレス（ROM 先頭コメント参照） |

#### 合否判定ロジック

```cpp
// 成功 → PC = $3469 で JMP *
// 失敗 → 別アドレスで JMP *（trap）
uint16_t last_pc = 0xFFFF;
while (true) {
    step();
    uint16_t pc = get_pc();
    if (pc == last_pc) {
        printf(pc == 0x3469 ? "PASS\n" : "FAIL at $%04X\n", pc);
        break;
    }
    last_pc = pc;
}
```

hoglet67/cpu_65c02.v は既に Dormann 65C02 テストを通過済み（README 記載）。

### 2.3 Woz Monitor インテグレーションテスト

`verif/wozmon/` に Verilator ネイティブの C++ インテグレーションテストがある。
Verilog RTL を Verilator でコンパイルし、wozmon ROM を埋め込んで自動実行する。

```bash
# ローカルで直接実行
verif/wozmon/run.sh

# Docker 経由（CI と同等環境）
docker compose -f docker/compose.yml run --rm wozmon
```

成功時の出力:

```text
test_wozmon:  13 PASS / 0 FAIL
test_programs: 34 PASS / 0 FAIL
```

#### テスト項目

| # | 内容 | 期待値 |
| --- | --- | --- |
| 1 | 起動プロンプト | `\` が表示される |
| 2 | `FF00` 単アドレス読み出し | `FF00: D8` |
| 3 | `FF00.FF0F` 範囲ダンプ | 2 行 × 8 バイトの hex dump |
| 4 | `0300: A9 55 4C 00 03` 書き込み | エコー確認 |
| 5 | `0300.0304` 読み戻し | `0300: A9 55 4C 00 03` |
| 6 | `0300: A9 55 85 00 A9 AA 85 01 00` 書き込み | エコー確認 |
| 7 | `0300.0308` ダンプ | 2 行（8+1 バイト）確認 |
| 8 | `0000.00FF` ゼロページ全ダンプ | 先頭・末尾行が `00` |
| 9 | `0300R` 実行 → ZP 書き込み確認 | `$00=55`, `$01=AA` |

#### 実装上の注意点

wozmon は通常コマンド後に `\` プロンプトを **再出力しない**。
起動時と ESC 時のみ `$FF1A`（ESCAPE ラベル）を経由して `\` が出力される。
通常コマンド後は `$FF1F` に直接ジャンプし、CR のみ出力して GETCHAR に戻る。
このため自動テストでは `\` を「コマンド完了」の合図として使えず、
結果文字列（`FF00: D8` など）の出現をもって合否判定する。

## 3. 手動動作確認

### 3.1 動作確認用テストプログラム集（wozmon 経由）

> **自動テスト対応**: `verif/wozmon/test_programs` が全 12 プログラムを自動検証する。
> IRQ ベクタを $FF00 に向け BRK 終端版プログラムを実行し、RAM 副作用で合否判定する。
> 手動実行時は Logic Analyzer / Register Panel で内部信号を観測しながら実行する。

wozmon 入力形式：`0300: AA BB CC ...` で書き込み → `0300R` で実行。

#### 3.1.1 最小確認：LDA ループ（5 bytes）

```text
0300: A9 55 4C 00 03
```

```asm
LDA #$55
JMP $0300
```

- Register Panel: A が常に `$55`
- Logic Analyzer: `sync` が 2 クロックおきに点灯、`addr` が `$0300→$0301→$0300…` と繰り返す

---

#### 3.1.2 メモリ Read/Write（9 bytes）

```text
0300: A9 AA 85 10 A5 10 4C 00 03
```

```asm
LDA #$AA
STA $10     ; WE=1、addr=$0010 へ書き込み
LDA $10     ; WE=0、addr=$0010 から読み込み
JMP $0300
```

- Logic Analyzer: `we=1` の赤セグメントが `$0010` に現れる
- Logic Analyzer: 次の `LDA` で同アドレスを `we=0` で読み返す

---

#### 3.1.3 スタック Push/Pop（11 bytes）

```text
0300: A9 11 48 A9 22 48 68 68 4C 00 03
```

```asm
LDA #$11
PHA          ; SP: FF→FE、$01FF に書き込み
LDA #$22
PHA          ; SP: FE→FD、$01FE に書き込み
PLA          ; SP: FD→FE、$01FE から読み込み（A=$22）
PLA          ; SP: FE→FF、$01FF から読み込み（A=$11）
JMP $0300
```

- Register Panel: SP が `FF→FE→FD→FE→FF` と変化する
- Logic Analyzer: `$01FF`・`$01FE` への `we=1/0` が交互に現れる

---

#### 3.1.4 フラグ・分岐（11 bytes）

```text
0300: A9 FF 69 01 B0 03 4C 10 03 4C 00 03
```

```asm
LDA #$FF
ADC #$01     ; A=$00、C=1、Z=1、N=0
BCS $030C    ; C=1 なので分岐する
JMP $0310    ; ここには来ない（来たら異常）
JMP $0300
```

- Register Panel: Flags の `C=1 Z=1` を確認
- Logic Analyzer: `BCS` 分岐ありサイクルで `sync` が飛ぶ

---

#### 3.1.5 インデックスアドレッシング（13 bytes + データ 5 bytes）

```text
0320: 11 22 33 44 55
0300: A2 00 BD 20 03 E8 E0 05 D0 F8 4C 00 03
```

```asm
; $0320 にデータを先に配置してから 0300R
LDX #$00
LDA $0320,X  ; X=0,1,2,3,4 と増加しながら読み出し
INX
CPX #$05
BNE $0302
JMP $0300
```

- Logic Analyzer: `addr` が `$0320→$0321→$0322→$0323→$0324` と順に現れる
- Register Panel: X が 0→1→2→3→4→0 とカウントする

---

#### 3.1.6 JSR / RTS（サブルーチン呼び出し、12 bytes）

```text
0300: 20 08 03 20 08 03 4C 00 03 E8 60
```

```asm
; $0300
JSR $0308    ; PC+2 をスタックに push（2 bytes）して $0308 へ
JSR $0308    ; 2 回目
JMP $0300
; $0308
INX          ; X をインクリメント
RTS          ; スタックから戻りアドレスを pop
```

- Register Panel: X が呼び出しごとに +1 される（`$01`→`$02`…）
- Logic Analyzer: JSR で `$01FE/$01FF` への write、RTS で read が見える
- Logic Analyzer: JSR は 6 サイクル、RTS は 6 サイクル（sync 間隔が長い）

---

#### 3.1.7 論理演算（AND / ORA / EOR、12 bytes）

```text
0300: A9 FF 29 0F 09 F0 49 AA 4C 00 03
```

```asm
LDA #$FF
AND #$0F     ; A=$0F（下位 4bit だけ残す）
ORA #$F0     ; A=$FF（上位 4bit をセット）
EOR #$AA     ; A=$55（XOR でビット反転）
JMP $0300
```

- Register Panel: A が `$FF→$0F→$FF→$55` と変化する
- Register Panel: N フラグが `1→0→1→0` と変化する

---

#### 3.1.8 シフト・ローテート（ASL / LSR / ROL / ROR、15 bytes）

```text
0300: A9 01 0A 0A 0A 4A 38 2A 6A 4C 00 03
```

```asm
LDA #$01
ASL A        ; A=$02  C=0
ASL A        ; A=$04  C=0
ASL A        ; A=$08  C=0
LSR A        ; A=$04  C=0
SEC          ; C=1 をセット
ROL A        ; A=$09  C=0（左ローテート、bit0 に C が入る）
ROR A        ; A=$04  C=1（右ローテート）
JMP $0300
```

- Register Panel: A のビットパターンと C フラグの連動を確認
- Logic Analyzer: 命令ごとに 2 サイクル（フェッチ + 実行）の sync 間隔

---

#### 3.1.9 ページクロス・ペナルティサイクル（13 bytes + データ）

```text
03FF: 42
0300: A2 01 BD FE 02 A0 01 B9 FE 02 4C 00 03
```

```asm
; $02FF: データ（アドレス $02FE + X=1 でページクロス発生）
LDX #$01
LDA $02FF,X  ; $0300 を読む（$02FF + 1 = $0300 = ページクロス）→ 5 サイクル
LDY #$01
LDA $02FF,Y  ; 同上 → 5 サイクル（ページクロスなしなら 4 サイクル）
JMP $0300
```

- Logic Analyzer: `LDA abs,X` がページクロス時に 5 サイクルになる（`sync` 間隔が 1 増える）
- Logic Analyzer: 無効読み出し（`$02FF` への余分なアクセス）が見える

---

#### 3.1.10 RMW 命令（INC / DEC メモリ、11 bytes）

```text
0300: A9 00 85 20 E6 20 E6 20 C6 20 4C 00 03
```

```asm
LDA #$00
STA $20      ; $20 = $00 に初期化
INC $20      ; $20 = $01（Read→Modify→Write の 5 サイクル）
INC $20      ; $20 = $02
DEC $20      ; $20 = $01
JMP $0300
```

- Logic Analyzer: `INC` 1 命令で `addr=$0020` への **read → write** が連続して見える（RMW）
- Logic Analyzer: write 直前に同アドレスへの無効 write が 1 サイクル挟まる（実チップ挙動）

---

#### 3.1.11 ゼロページ X インデックス（12 bytes）

```text
0050: 11 22 33 44
0300: A2 00 B5 50 E8 E0 04 D0 F9 4C 00 03
```

```asm
; $0050〜$0053 にデータを先に配置
LDX #$00
LDA $50,X    ; ZP + X でアクセス（$50〜$53 を順に読む）
INX
CPX #$04
BNE $0302
JMP $0300
```

- Logic Analyzer: アドレスが `$0050→$0051→$0052→$0053` と ZP 内で移動する
- Logic Analyzer: ZP アドレスのため絶対アドレス指定より 1 バイト短く、1 サイクル速い（4 サイクル）

---

#### 3.1.12 間接インデックス（(zp),Y アドレッシング、11 bytes）

```text
0400: 55
0020: 00 04
0300: A0 00 A9 00 91 20 C8 C0 04 D0 F9 4C 00 03
```

```asm
; $0020/$0021 にポインタ $0400 を先に配置
; $0400 をクリアバッファとして使用
LDY #$00
LDA #$00
STA ($20),Y  ; ($20) = $0400、Y=0 → $0400+0 に書き込み
INY
CPY #$04
BNE $0302
JMP $0300
```

- Logic Analyzer: ZP 読み出し（`$0020`・`$0021`）→ ポインタ参照（`$0400+Y`）の 2 段アクセスが見える
- Logic Analyzer: 間接アドレス解決で合計 6 サイクル

### 3.2 Integer BASIC スモークテスト

> **自動テスト対応**: `verif/wozmon/test_programs` §3.2 が自動検証する。
> `E000R` → `PRINT 2+2` → 出力に `4` が含まれることを確認する。

harness.cpp に Integer BASIC ROM（$E000–$EFFF）が組み込まれている。
wozmon から `E000R` で起動でき、人間が目で結果を確認できる最もわかりやすいテスト。

```text
1. シミュレータ起動 → \ プロンプト確認
2. E000R → BASIC 起動（プロンプト表示なし）
3. PRINT 2+2  → 4  ← 自動検証済み
4. PRINT 10*10 → 100
5. PRINT 32767+1 → オーバーフロー挙動の確認
```

**検証できること**: 乗算・除算・加算ルーチン、文字列出力、入力ループ

## 4. 将来テスト計画

### 4.1 Tom Harte ProcessorTests（サイクル精度）

1 命令ごとに初期状態・期待最終状態・メモリアクセス列を JSON で定義した
10,000 ケース/opcode のテストスイート。

#### 検証内容

- 各サイクルのアドレスバス（AB）・データバス（DB）・WE の値
- 実行後のレジスタ（A, X, Y, SP, PC, P）
- 実行後のメモリ書き換え内容

#### 実施方針

```text
verif/tb_unit/ に C++ テストドライバを作成
  → JSON を読み込み、サイクルごとに AB/DB/WE を eval() 後に比較
  → 失敗時はサイクル番号・期待値・実測値を出力
```

**実装コスト**: 中（JSON パーサが必要、nlohmann/json 等を利用）

**参照**: <https://github.com/TomHarte/ProcessorTests/tree/main/nes6502>
（65C02 版も同リポジトリに存在）
