# Apple-II soft-FPGA 化の労力検討

Apple-I の soft-FPGA 実装を基準に、Apple-II への拡張コストを整理する。

## 1. 共通部分（再利用できる）

| コンポーネント | 状況 |
|--------------|------|
| 6502 RTL コア | そのまま流用 |
| C++ ハーネス・リングバッファ | 基本構造は流用 |
| Emscripten / Wasm パイプライン | そのまま |
| xterm.js コンソール | テキストモードで継続利用 |

---

## 2. 差分コストの分類

### 小（Apple-I 比 1〜2 倍）

#### メモリマップの拡張

- RAM: 4 KB → 48 KB
- ソフトスイッチ（`$C000–$C0FF`）の追加
- ページ切り替え・言語カード（16 KB バンク）

#### キーボード

- Apple-I は PIA（6821）経由。Apple-II は専用エンコーダ（AY-3-4600）
- soft-FPGA では JS → Wasm キー注入で抽象化できるため差は小さい

#### サウンド

- 1bit スピーカー（`$C030` をアクセスするたびにトグル）
- Web Audio API で矩形波を出すだけ

---

### 中（3〜5 倍）

#### テキストモード描画

- 40列 × 24行、非線形メモリレイアウト（`$0400, $0480, $0500…` と行順が飛ぶ）
- キャラクタ ROM（2513 相当）をフォントデータとして実装
- Canvas への文字描画ルーティン

#### ローレゾグラフィックス（Lo-Res）

- 40 × 48、16色、`$0400–$07FF`（テキストページと共有）
- ソフトスイッチで Text / Lores / Mixed を切り替え

#### Applesoft BASIC / Integer BASIC ROM

- ROM バイナリ自体は公開されているので組み込みは容易
- Applesoft は 10 KB あり、デバッグ量が増える

---

### 大（8〜15 倍）

#### ハイレゾグラフィックス（Hi-Res）

- 280 × 192、6色（NTSC アーティファクト色）
- メモリレイアウトが複雑（`$2000` 起点、8行ごとに 40 バイト飛ぶビットインタリーブ）
- NTSC カラー生成はブラウザ Canvas では近似実装が必要

#### Disk II（Woz フロッピーコントローラ）

- 自己同期型フロッピー制御の RTL 再現が難しい
- DSK / NIB イメージのロード・トラックビット列エミュレーションが必要
- これがないと DOS 3.3 も ProDOS も動かない

## 4. 優先度の考察

Apple-II はコンポーネントの**量的拡張**であり、
Apple-I で実証した「バスサイクル・PIA ハンドシェイクの可視化」という核心価値の
**新しい側面を加えるものではない**。

プロジェクトのロードマップ上は、異なるアーキテクチャ（4004 / Z80）の方が
可視化ショーケースとしての価値が高い可能性がある。

Apple-II を選ぶ明確な理由があるとすれば:

- Applesoft BASIC でのデモ体験（`GR` / `HGR` グラフィック）
- NTSC カラーアーティファクトという独特の可視化対象
- Apple-I → Apple-II という歴史的連続性の展示
