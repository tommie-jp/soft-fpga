# 11. vm80a タイミング解析メモ

vm80a_core（Intel 8080A RTL クローン）を Verilator でシミュレーションした際に
判明したタイミング詳細を記録する。
cpm_top.v の I/O 制御設計の根拠として参照すること。

---

## 1. f1/f2 二相クロックの位相

cpm_top.v では f1/f2 を **レジスタ** で生成している。

```verilog
always @(posedge clk) begin
    f1 <= ~f1;
    f2 <=  f1;   // f2 は f1 の 1 クロック遅延
end
```

リセット直後の初期値は f1=0, f2=0（Verilog の reg 初期値）。
最初の posedge の後に f1=1, f2=0 となる。

vm80a_core は **posedge clk** でラッチし、f1/f2 として **その posedge 直前の値**（OLD 値）を参照する。

| シミュレーションサイクル c | vm80a が見る f1 | vm80a が見る f2 | 意味 |
|:---:|:---:|:---:|:---|
| 0 | 0 | 0 | リセット |
| 1 | 1 | 0 | f1 アクティブ（T ステート後半） |
| 2 | 0 | 1 | f2 アクティブ（T ステート前半） |
| 3 | 1 | 0 | f1 アクティブ |
| … | … | … | … |

> **結論**: **奇数 c = f1 アクティブ（T ステート後半）、偶数 c = f2 アクティブ（T ステート前半）**

Intel 8080 のデータシートでは:

- **Φ2（f2）** が立ち上がった時点が T ステートの境界
- **Φ1（f1）** は T ステート後半のアクション（レジスタ更新など）に使う

---

## 2. MVI A, nn 命令のマシンサイクル構造

Intel データシート上は **7 T ステート** だが、vm80a 内部では以下の 4 マシンサイクルを実行する。

| マシンサイクル | T 数 | 内容 |
|:---:|:---:|:---|
| M1 | 4T | 命令フェッチ（SYNC サイクル、MVI A = 0x3E） |
| M2 | 3T | 即値バイト読み出し（ZP レジスタ r16_wz[7:0] に格納） |
| M3 | 3T | ダミーメモリリードサイクル |
| M4 | 3T | レジスタ更新（acc ← 即値） |

**合計 13 T ステート**（データシートより多い。vm80a 固有の動作）。

### acc 更新タイミング

```verilog
// vm80a.v より
assign a357 = ...
           | acc_sel & id08 & (t5 & m1 | t3 & m4);
```

`acc_sel = 1`（i[5:3]=111 = レジスタ A 指定）かつ `id08 = 1`（MVI A は id08 に含まれる）の場合、
**M4 T3 f1**（奇数サイクル）で `alu_awr = 1` となり、`acc ← d` が実行される。

M4 T3 f1 が確定するのは d-bus が安定している M4 T3 の posedge。
このとき `dbin_ext=1` を通じて `di`（即値バイト）が d-bus に再出力されている。

## 3. OUT port, A 命令のマシンサイクル構造

Intel データシート上は **10 T ステート** で、vm80a も同じ。

| マシンサイクル | T 数 | 内容 |
|:---:|:---:|:---|
| M1 | 4T | 命令フェッチ（SYNC サイクル、OUT = 0xD3） |
| M2 | 3T | ポート番号バイト読み出し |
| M5 | 3T | I/O 書き込み（vm80a では M3 ではなく M5 を使う） |

> vm80a では I/O 書き込みサイクルを **M5** と呼ぶ（メモリ書き込みが M3）。

### ステータス信号

| 信号 | 値 | タイミング |
|:---|:---:|:---|
| `sy_m1` | 1 | M1 期間中 |
| `sy_out` = `m5 & id_out` | 1 | M5 期間中（OUT 命令） |
| `sy_wo_n` | 0 | M5 期間中（`id_io` が id85 に含まれるため） |

SYNC サイクル（M1 T1 f2）で `cpu_dout` に出力されるステータスバイトの意味:

- `0xA2` : FETCH（命令フェッチ）
- `0x82` : メモリリード M2
- `0x10` : OUT サイクル M5（bit4 = OUT フラグ）

### アキュムレータの xr レジスタへのコピー

```verilog
assign alu_xwr = a327 | t4f1 & (id_rst | id_out | ~id_11x);
```

OUT 命令の M1 T4 f1（奇数サイクル）で `alu_xwr = 1` となり、`xr ← acc` が実行される。
M5 で I/O 書き込みが行われる時点では `cpu_dout` = `xr` の値になっている。

## 4. cpu_dout と dbg_a の対応

cpm_top.v では `dbg_a = cpu_dout`（vm80a の `pin_dout` = `cpu_dout`）を観測している。

| サイクル | cpu_dout の内容 |
|:---|:---|
| SYNC サイクル | ステータスバイト（0xA2, 0x82, 0x10 など） |
| M1 T4 以降（OUT） | xr の値（= acc のコピー） |
| M5 T2 f2（OUT） | xr の値（wr_n が Low の期間） |

ハーネスの `dbg_a` 出力で観測する場合、**ステータスバイト 0x10 の後に正しいデータバイトが続く**ことを確認する。

## 5. バグ解析と修正（解決済み）

### 5.1 io_req 二重発火バグ（修正済み）

cpm_top.v の io_req 生成ロジックに **wr_n の立ち下がりエッジを検出する条件が欠如**していた。
立ち上がりエッジ検出のみに修正し、OUT ごとに 1 回だけ発火するようにした。

```verilog
// 修正後のコード（cpm_top.v）
always @(posedge clk) begin
    prev_wr_n <= cpu_wr_n;
    io_req    <= 1'b0;
    if (cycle_out && !prev_wr_n && cpu_wr_n) begin  // 立ち上がりエッジのみ
        io_req  <= 1'b1;
        io_wr   <= 1'b1;
        io_addr <= cpu_addr[7:0];
        io_dout <= cpu_dout;
    end
end
```

### 5.2 リセット時 RAM 上書きバグ（根本原因・修正済み）

**症状**: 1 回目の OUT で常にデータ `0x00` が出力される。

**根本原因**:

Verilator では、`reg` のリセット初期値はすべて `0` である。
これにより、シミュレーション起動直後は `cpu_wr_n = 0`（ライトイネーブル = アクティブ）かつ
`cpu_addr = 0x0000` の状態になっている。

ハーネスの `sim_test()` では以下の順序で初期化を行っていた:

1. `top = new Vcpm_top` — 全シグナル初期値 0 (`wr_n=0`, `addr=0`)
2. `top->cpm_top->ram[0] = 0x3E` — プログラム配置（`MVI A,'H'`）
3. `top->reset = 1` → `posedge clk` 評価

**ステップ 3 の最初の posedge で** cpm_top.v の RAM ライトロジックが:

```verilog
always @(posedge clk) begin
    if (!cycle_io && !cpu_wr_n)    // cpu_wr_n=0（初期値）→ 条件成立！
        ram[cpu_addr] <= cpu_dout; // ram[0] <= 0x00（cpu_dout初期値）
end
```

を実行し、`ram[0x0000] = 0x00` に上書きしてしまう。

CPU は MVI A（0x3E）ではなく NOP（0x00）から実行を開始するため:

| 実際に実行された命令 | アドレス | バイト |
|:---|:---:|:---:|
| NOP（初期値0x00が上書きされたため） | 0x0000 | 0x00 |
| MOV C,B（即値バイト0x48をオペコードとして誤解釈） | 0x0001 | 0x48 |
| OUT 0x01（0xD3 = OUT opcode） | 0x0002 | 0xD3 |

OUT 命令時、acc=0x00（MVI A 未実行）のため OUT は 0x00 を送出する。
2 回目の MVI A（0x0004: 0x3E）は上書きされていないため正常に実行され、acc=0x69='i'。

**修正方法**: cpm_top.v の RAM ライト条件に `!reset` を追加。

```verilog
// 修正後のコード（cpm_top.v）
always @(posedge clk) begin
    if (!reset && !cycle_io && !cpu_wr_n)
        ram[cpu_addr] <= cpu_dout;
end
```

外部 `reset` 入力が HIGH の間は RAM への書き込みをブロックするため、
Verilator 初期値に起因するスプリアスライトが発生しなくなる。

**修正後の動作確認**:

```bash
./build/sim --test
# [test] sim_test: 2 bytes out (Hi)  ← PASS
```

## 6. sim_test ハーネス概要

[`examples/06-8080/cxx/harness.cpp`](../../examples/06-8080/cxx/harness.cpp) の `sim_test()` 関数が
単体テスト用エントリポイントになっている。

```cpp
// テストプログラム（0x0000 から配置）
static const uint8_t prog[] = {
    0x3E, 'H',   // MVI A, 'H'     0x0000
    0xD3, 0x01,  // OUT $01 CONOUT 0x0002
    0x3E, 'i',   // MVI A, 'i'     0x0004
    0xD3, 0x01,  // OUT $01        0x0006
    0x76,        // HLT             0x0008
};
```

実行方法:

```bash
cd examples/06-8080
cmake -B build && cmake --build build
./build/sim --test    # 戻り値 0 = PASS（出力バイト数 >= 2）
```

デバッグトレースはビルド時に埋め込まれており `stderr` に出力される。
